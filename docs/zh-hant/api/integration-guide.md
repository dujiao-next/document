# 站點對接說明

> 更新時間：2026-03-01

本文用於說明 Dujiao-Next 站點之間如何進行對接與串貨，協助站長快速跑通從「開通權限」到「下單採購」的完整流程。

## 1. 先理解兩個角色

- 對接方（A 站）：拉取上游商品、在自己站點售賣，並在使用者下單後向上游發起採購。
- 被對接方（B 站）：開放介面給其他站點呼叫，接收採購請求並從對接使用者錢包扣款。

## 2. 標準流程（A 對接 B）

1. B 站使用者在前台開通 API 權限，生成 Access Key / Secret。
2. A 站管理員在後台新增「對接連線」，填寫 B 站地址與憑證。
3. A 站測試連線成功後，拉取 B 站商品並建立商品映射。
4. A 站使用者下單後，A 站會向 B 站發起採購單。
5. B 站按規則處理訂單並扣減該對接使用者的錢包餘額。

## 3. 被對接方（B 站）需要做什麼

### 3.1 設定開放介面地址

後台路徑：`系統設定 -> 品牌與站點`，設定「開放介面地址（Open API URL）」，例如：

- `https://api.example.com/api/open/v1`

這個地址會展示給前台使用者，用於提供給對接方呼叫。

### 3.2 使用者開通 API 權限並生成 Token

前台路徑：`個人中心 -> API 權限`

使用者可執行以下操作：

- 開通 API 權限
- 建立多個 Token（Access Key / Secret）
- 設定 Token 到期時間
- 啟用/停用或刪除 Token

注意：`Secret` 只在建立時展示一次，請立即保存。

## 4. 對接方（A 站）如何新增連線

後台路徑：`站點對接 -> 對接連線 -> 新增`

建議填寫規則：

| 欄位 | 填寫說明 |
| --- | --- |
| 連線名稱 | 自訂，方便區分來源站點 |
| 站點地址 | 填 B 站根地址，如 `https://b.example.com` |
| 協議類型 | 使用預設值（Dujiao OpenAPI v1） |
| 協議版本 | 使用預設 `v1` |
| Access Key | B 站使用者生成的 Access Key |
| Secret | B 站使用者生成的 Secret |
| 回調地址 | 可選，建議填 A 站可公開存取地址 |

注意：

- 「站點地址」建議填根地址，不要手動拼 `/api/open/v1`，系統會按協議固定路徑呼叫。

## 5. 拉取商品與建立映射

後台路徑：`站點對接 -> 商品映射`

建議流程：

1. 先選擇連線並拉取上游商品。
2. 選擇本地商品與上游商品/SKU 的映射關係。
3. 依你的營運策略設定本地價格與展示資訊。
4. 啟用映射後再對外售賣。

## 6. 下單與扣款行為

- A 站使用者下單後，系統會自動向 B 站建立採購單。
- B 站會從該對接使用者錢包扣款。
- 若餘額不足，採購會立即失敗並回傳錯誤，不會延遲扣款。
- 若庫存不足或商品不可用，也會立即回傳失敗。

說明：

- 未啟用「餘額不足自動停售」時，映射商品不會自動下架，但訂單會以失敗狀態返回。
- 前台使用者不會看到「第三方對接來源」標識，來源資訊僅後台可見。

## 7. 營運排查建議

### A 站（對接方）重點看

- `站點對接 -> 對接訂單`：查看採購狀態、失敗原因、重試情況。
- `站點對接 -> 回調日誌`：查看上游狀態回傳是否成功。

### B 站（被對接方）重點看

- `站點對接 -> 被對接使用者`：查看哪些使用者已開通 API 權限、Token 使用情況。
- `站點對接 -> 對接訂單`：篩選 `source=inbound` 查看外部站點採購單。

## 8. 常見問題

### Q1：測試連線回傳 401

優先檢查：

- Access Key / Secret 是否填錯
- Token 是否已停用或過期
- 站點地址是否可公開存取

### Q2：下單時報參數或簽名錯誤

優先檢查：

- 請求是否帶簽名頭
- 寫介面是否帶 `X-DJ-Idempotency-Key`
- 時間戳是否偏差過大

### Q3：使用者下單成功但採購失敗

常見原因：

- B 站錢包餘額不足
- 上游庫存不足
- 商品映射關係錯誤（商品 ID / SKU ID 不匹配）
